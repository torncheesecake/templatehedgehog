# Template Hedgehog – Technical Proof Section Blueprint

This document defines the exact layout, composition, and intent of the homepage technical proof section.

Codex MUST follow this. Do not improvise.

---

## PURPOSE

This is the flagship trust section.

Its job is to make a developer think:

“I can see exactly what I’m getting.”

This section should prove:
- source exists
- compiled output exists
- workflow mapping exists
- pack structure exists

---

## SECTION ROLE

This should feel like:
- one composed product proof block
- a premium technical feature
- a strong trust-building moment

It should NOT feel like:
- several separate proof cards
- a collection of boxes
- a mixed content dump

---

## CONTENT STRUCTURE

### Intro
- Eyebrow (optional)
- Strong heading
- One short support line

### Main proof block
Two-column split:

#### Left side
**MJML → Compiled HTML**
- show source snippet
- show compiled output snippet
- clearly framed as transformation

#### Right side
**Pack structure + workflow mapping**
- file tree excerpt
- mapping path / stack path
- one supporting visual or structured list

---

## LAYOUT

### Desktop
- Wide split layout
- Left side slightly dominant if needed
- Right side grouped as one supporting proof column

### Tablet
- Keep split if space allows
- Otherwise stack with MJML/HTML first

### Mobile
- Intro first
- MJML/HTML block second
- Pack structure + mapping third

---

## VISUAL COMPOSITION

The whole section must read as ONE feature.

Use:
- shared parent composition
- aligned top edges
- consistent inner spacing
- one clear rhythm

Do NOT:
- present three or four disconnected cards
- create too many nested boxes
- overuse borders

---

## LEFT SIDE RULES

The MJML and HTML examples should feel paired.

They should not look like random code blocks.

Use:
- clear labels
- aligned framing
- enough spacing to read comfortably
- strong visual link between source and output

---

## RIGHT SIDE RULES

Pack structure and mapping must feel product-like.

Use:
- structured file tree
- concise mapping explanation
- one supporting visual or ordered path

Avoid:
- clutter
- too many tiny sub-panels
- excessive border noise

---

## DESIGN LANGUAGE

- Technical but premium
- Clean
- Calm
- Product-focused
- More feature section than documentation block

---

## FAILURE CONDITIONS

The section is wrong if:
- it still feels like several boxes placed next to each other
- the code blocks feel cramped
- the right side looks like miscellaneous leftovers
- there is no clear visual link between source, output, and mapping
- the section does not feel materially stronger than before
